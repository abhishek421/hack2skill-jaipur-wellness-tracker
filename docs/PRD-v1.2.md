# Product Requirements Document

# MindTrack — AI-Generated Weekly Narrative

Version: 1.2
Type: Feature Update
Status: Ready for Development
Parent: PRD v1.0 (Wellness Actions)

---

## Overview

Replace the rule-based weekly narrative string with an LLM-generated paragraph that reads the student's full week of emotional data and produces a personalized, empathetic, human-quality summary.

The narrative is the most visible text output in the Weekly Summary screen. Upgrading it from a template sentence to an AI-written reflection significantly raises perceived product quality and emotional resonance — without requiring changes to check-in flow, database schema, or authentication.

---

## Problem With Current Implementation

The existing weekly narrative is a single hardcoded template string:

> "This week, exam anxiety remained your most common stress trigger. Your mood remained stable, but energy levels decreased during the final three days."

This sentence is generated by substituting top-trigger and trend variables into a fixed pattern. It has three structural problems:

1. **No empathy.** It reads like a report, not a reflection. Students in distress respond poorly to clinical summaries.
2. **No depth.** It ignores reflection journal content, co-occurring triggers, day-over-day nuance, and emotional trajectory.
3. **No individuality.** Two students with identical averages get identical text even if their journeys were completely different.

---

## Goal

Generate a 3–5 sentence weekly narrative using an LLM that:

- Reads like it was written by a supportive peer or counsellor
- References the student's actual data (trigger names, mood shifts, journal themes)
- Acknowledges difficulty without dramatising it
- Ends with one forward-looking, encouraging observation
- Stays non-clinical and non-diagnostic at all times

---

## Scope

### In Scope

- New server-side function: `generateWeeklyNarrative(data)`
- Updated `/api/weekly-summary` route to call this function and return AI narrative
- Graceful fallback to rule-based narrative if LLM call fails
- Prompt design, input construction, output parsing
- Display on Weekly Summary page (no UI layout changes, text slot already exists)
- Caching: store generated narrative in database to avoid re-calling LLM on page reload

### Out of Scope

- Journal sentiment analysis (separate feature)
- Real-time narrative during check-in
- Streaming / typewriter display effect (nice-to-have, post-MVP)
- Per-user model configuration
- Narrative history across multiple weeks (future)

---

## User Story

**As** a student who has completed at least three check-ins this week,
**When** I open my Weekly Summary,
**I want** to read a narrative that feels written for me specifically,
**So that** I feel understood and motivated to keep tracking.

## Functional Requirements

### FR-1: Narrative Generation Function

A server-side TypeScript function `generateWeeklyNarrative(input: WeeklyNarrativeInput): Promise<string>` must be implemented.

**Input shape:**

```ts
type WeeklyNarrativeInput = {
  studentName: string;
  weekStartDate: string;           // ISO date, e.g. "2026-05-31"
  checkInCount: number;            // 0–7
  averageMood: number;             // 1–5 float
  averageStress: number;           // 1–5 float
  averageEnergy: number;           // 1–5 float
  moodTrend: 'improving' | 'declining' | 'stable' | 'volatile';
  stressTrend: 'improving' | 'declining' | 'stable' | 'volatile';
  topTriggers: string[];           // e.g. ["Exam Anxiety", "Poor Sleep"]
  triggerFrequency: Record<string, number>;  // e.g. { "Exam Anxiety": 4 }
  reflectionSnippets: string[];    // Up to 3 anonymised excerpts, max 100 chars each
  lowestMoodDay: string | null;    // ISO date of worst mood day
  highestMoodDay: string | null;   // ISO date of best mood day
};
```

**Output:** Plain text string, 3–5 sentences, no markdown, no bullet points.

---

### FR-2: LLM Provider

Use the Anthropic Claude API (`claude-haiku-4-5-20251001`) as the default model.

- Haiku is fast (<1s), cheap (~$0.00025 per call at typical weekly summary token counts), and sufficient for 3–5 sentence generation.
- Model is configurable via environment variable `NARRATIVE_MODEL` to allow upgrade without code changes.
- Use the official `@anthropic-ai/sdk` npm package.

---

### FR-3: Prompt Design

The system prompt must:

1. Establish the AI's persona as a warm, non-clinical peer supporter
2. Explicitly prohibit diagnostic language (no "you may have anxiety", "you seem depressed")
3. Explicitly prohibit advice (recommendations are handled by the existing engine)
4. Instruct the model to use the student's first name once
5. Instruct the model to acknowledge specific triggers by name
6. Instruct the model to reference journal themes if available
7. Instruct the model to end on a forward-looking note
8. Cap output at 120 words

**System prompt (exact):**

```
You are a warm, supportive peer writing a short weekly reflection for a student.
Write 3 to 5 sentences in plain English. Do not use bullet points, headers, or markdown.
Do not diagnose. Do not prescribe. Do not use clinical language.
Do not give advice — a separate recommendation engine already handles that.
Reference the student's actual data: their name, triggers, mood pattern, and journal themes if provided.
Use the student's first name once, naturally.
End with one forward-looking sentence that is encouraging without being dismissive of difficulty.
Keep the total response under 120 words.
```

**User message structure:**

```
Student name: {name}
Week: {weekStartDate} (7 days)
Check-ins completed: {checkInCount}/7
Average mood: {averageMood}/5
Average stress: {averageStress}/5
Average energy: {averageEnergy}/5
Mood trend this week: {moodTrend}
Stress trend this week: {stressTrend}
Top stress triggers: {topTriggers.join(', ')}
Trigger frequency: {JSON.stringify(triggerFrequency)}
Journal themes (excerpts): {reflectionSnippets.join(' | ')}
Lowest mood day: {lowestMoodDay}
Best mood day: {highestMoodDay}
```

---

### FR-4: API Route Update — `/api/weekly-summary`

Current behaviour: returns static narrative string from rule engine.

New behaviour:

1. Assemble `WeeklyNarrativeInput` from existing weekly query results (no new DB queries needed)
2. Call `generateWeeklyNarrative(input)`
3. Return narrative in response alongside existing metrics
4. On LLM error: catch, log error, fall back to existing rule-based narrative string
5. Store generated narrative in `weekly_summaries` table (see FR-6)

Response field addition:

```ts
{
  // ... existing fields unchanged ...
  narrative: string;           // AI-generated or rule-based fallback
  narrativeSource: 'ai' | 'rule-based';   // for debugging/monitoring
}
```

---

### FR-5: Fallback Behaviour

If the LLM call fails for any reason (network error, API key missing, rate limit, timeout):

- Log the error server-side with `console.error`
- Return the existing rule-based narrative string
- Set `narrativeSource: 'rule-based'` in response
- Do not surface the error to the student

The student experience must never degrade because of an AI failure.

---

### FR-6: Caching — Weekly Summary Storage

Avoid calling the LLM repeatedly for the same week.

Add a `weekly_summaries` table:

```sql
CREATE TABLE weekly_summaries (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  week_start   DATE NOT NULL,
  narrative    TEXT NOT NULL,
  source       TEXT NOT NULL CHECK (source IN ('ai', 'rule-based')),
  generated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (user_id, week_start)
);
```

On `/api/weekly-summary` request:

1. Check if a row exists for `(user_id, week_start)`
2. If yes: return cached narrative (skip LLM call)
3. If no: generate, store, return

Cache is intentionally permanent per week — the input data doesn't change once the week closes.

For the current (open) week: regenerate on each request, do not cache, since check-ins may still be added.

---

### FR-7: Environment Variable

Add to `.env.local` and Vercel project settings:

```
ANTHROPIC_API_KEY=sk-ant-...
NARRATIVE_MODEL=claude-haiku-4-5-20251001
```

`NARRATIVE_MODEL` defaults to `claude-haiku-4-5-20251001` if not set.

---

### FR-8: Input Sanitisation

Before passing reflection snippets to the LLM:

- Strip all content longer than 100 characters per snippet (truncate with `…`)
- Limit to 3 snippets maximum (most recent)
- Remove any PII patterns: email addresses, phone numbers (regex strip)
- Escape curly braces to prevent prompt injection via journal content

---

## Non-Functional Requirements

### Latency

- LLM call must complete within 5 seconds
- Set a 5s timeout on the Anthropic SDK call
- If timeout exceeded, trigger fallback immediately

### Cost

- Estimated token usage per call: ~400 tokens input + ~120 tokens output = ~520 tokens
- At Haiku pricing: <$0.001 per weekly summary
- At 1,000 active users/week: <$1/week

### Privacy

- Reflection content is sent to the Anthropic API
- This must be disclosed in the product's privacy policy
- Reflection snippets must be truncated and anonymised before sending (see FR-8)
- No student name, email, or user ID is included in the prompt — only first name

### Reliability

- Fallback ensures 100% uptime for the Weekly Summary feature regardless of LLM status
- No single point of failure introduced

## Data Flow

```
GET /api/weekly-summary
        │
        ▼
Fetch week's check_ins, triggers, reflections from Supabase
        │
        ▼
Check weekly_summaries cache (closed weeks only)
        │
   ┌────┴────┐
Cached     Not cached / open week
   │               │
   ▼               ▼
Return         Assemble WeeklyNarrativeInput
cached              │
narrative           ▼
              Call generateWeeklyNarrative()
                    │
            ┌───────┴───────┐
          Success          Failure
            │                │
            ▼                ▼
     Store in cache    Rule-based fallback
            │                │
            └───────┬────────┘
                    ▼
           Return narrative + source flag
                    │
                    ▼
          Weekly Summary page renders
```

## Example Output

**Input:** Ananya, 5 check-ins, avg mood 2.8, avg stress 4.1, top triggers: Exam Anxiety + Self-Doubt, declining mood trend, journal mentions "don't feel ready"

**Generated narrative:**

> Ananya, this was a genuinely difficult week — your stress stayed high and your mood gradually dipped as the week progressed. Exam anxiety and self-doubt showed up together most days, and your reflections suggest that feeling unprepared was weighing on you heavily. It makes sense that energy took a hit alongside all of that. The fact that you kept checking in through it all is worth acknowledging — five days of showing up, even when things felt hard. Next week, even one lighter day can shift the pattern.

**Input:** Rohan, 7 check-ins, avg mood 3.9, avg stress 2.4, top trigger: Study Backlog, stable mood, improving energy

Study backlog was your main stressor, but it didn't drag your mood down the way it sometimes can. Your reflections pointed toward staying focused, which tracks with how the numbers look. Finishing the week with seven check-ins is a real streak — carry that consistency into next week.

## Acceptance Criteria

| # | Criterion |
|---|-----------|
| AC-1 | Weekly Summary page displays AI-generated narrative for students with ≥3 check-ins |
| AC-2 | Narrative references at least one of the student's actual triggers by name |
| AC-3 | Narrative uses the student's first name exactly once |
| AC-4 | Narrative is between 40 and 120 words |
| AC-5 | No diagnostic or clinical language appears in any generated narrative |
| AC-6 | If `ANTHROPIC_API_KEY` is absent, weekly summary still loads with rule-based narrative |
| AC-7 | If LLM call times out (>5s), fallback narrative is returned, no error shown to student |
| AC-8 | Closed-week narratives are cached; same student requesting same week returns same text |
| AC-9 | Open-week narratives are never cached |
| AC-10 | `narrativeSource` field in API response correctly reflects `'ai'` or `'rule-based'` |

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| LLM returns clinical/harmful language | Low | System prompt explicitly prohibits; test with distress-signal inputs |
| API key not set in production | Medium | Fallback to rule-based; add key check to startup logs |
